# گزینه‌های پیشرفته (هر گزینه چه چیزی را تغییر می‌دهد؟)

این صفحه گزینه‌های «پیشرفته»‌ای را که در UI دسکتاپ و برنامهٔ اندروید می‌بینید توضیح می‌دهد و می‌گوید تغییر هرکدام چه اثری روی **سرعت**، **پایداری**، **مصرف سهمیه (quota)** و **اعتماد/امنیت** دارد.

اگر تازه‌کارید: پیش‌فرض‌ها را نگه دارید و فقط وقتی یک **علائم مشخص** دارید سراغ این‌ها بروید (timeout، جهش 504، تمام شدن quota، بافر/کندی یوتیوب، یا گوگل‌آی‌پی ناپایدار).

---

## گزینه‌های کارایی / سرعت

### `runtime_auto_tune` + `runtime_profile`

- **چی هست؟** وقتی فعال باشد، برنامه برای چند گزینهٔ مهم (مثل رِنج‌پارالل، اندازهٔ چانک، timeoutها و گاهی fan-out) بر اساس یک «پروفایل» مقدار پیشنهادی انتخاب می‌کند.
- **اثر روی تجربهٔ کاربر**:
  - **eco**: همزمانی کمتر → مصرف quota کمتر، معمولاً پایدارتر روی اینترنت ضعیف، ولی کندتر برای دانلودهای بزرگ
  - **balanced**: تعادل معمول و پیش‌فرض
  - **max_speed**: همزمانی/timeoutها برای throughput → سریع‌تر، ولی می‌تواند quota را سریع‌تر بسوزاند و اگر ظرفیت کم باشد جهش‌های 504 را بدتر کند
- **کی تغییر بدهم؟**
  - اگر نمی‌خواهید دستی تنظیم کنید، روشنش کنید.
  - وقتی quota مهم است یا مسیر شما شکننده است، **eco** بهتر است.
  - **max_speed** فقط وقتی معنی دارد که چند `script_id`/اکانت دارید و ظرفیت دارید.

### `parallel_relay`

- **چی هست؟** برای هر درخواست منطقی، چند رلهٔ Apps Script را همزمان می‌زند و اولین پاسخ موفق را برمی‌گرداند.
- **اثر روی UX**:
  - مقدار بالاتر می‌تواند «گیرکردن‌های طولانی» یک اسکریپت کند را کم کند → مرور وب روان‌تر
  - ولی تعداد فراخوانی‌ها را چند برابر می‌کند → quota سریع‌تر مصرف می‌شود و روی صفحات سنگین می‌تواند burst storm بسازد
- **قاعدهٔ سریع**:
  - `0`/`1`: حالت عادی و کم‌مصرف
  - `2`: بهبود ملایم برای اسکریپت‌های ناپایدار
  - `3+`: فقط اگر واقعاً quota/ظرفیت اضافه دارید

### `relay_rate_limit_qps` + `relay_rate_limit_burst`

- **چی هست؟** محدودکنندهٔ نرم (token bucket) برای صاف‌کردن جهش‌های رله.
- **اثر روی UX**:
  - QPS کمتر → جهش کمتر و معمولاً 504 کمتر، ولی ممکن است صفحات «کندتر» حس شوند چون درخواست‌ها صف می‌شوند
  - burst بیشتر → شروع سریع‌تر صفحه، ولی همچنان می‌تواند quota را بکوبد
- **کی تغییر بدهم؟**
  - وقتی الگوی «اوج → خطا → بهبود» دارید، فعال کردنش کمک می‌کند.
  - برای بافر/کندی ویدیو که ناشی از تعداد درخواست‌هاست هم می‌تواند مفید باشد.

### `range_parallelism` + `range_chunk_bytes`

- **چی هست؟** برای دانلودهای بزرگ (GET)، برنامه می‌تواند با **range-parallel** کار کند: یک probe با `Range` می‌زند، سپس بقیهٔ چانک‌ها را موازی می‌گیرد و در نهایت برای کلاینت یک `200 OK` کامل می‌سازد.
- **اثر روی UX**:
  - parallelism بالاتر → سریع‌تر، ولی تعداد تماس همزمان بیشتر → فشار quota و احتمال خطا بیشتر
  - chunk بزرگ‌تر → تعداد تماس کمتر (quota-friendly)، ولی هر تماس طولانی‌تر می‌شود و به محدودیت‌های Apps Script نزدیک‌تر
- **قاعدهٔ سریع**:
  - اگر مشکل quota است: **چانک بزرگ‌تر** + **parallelism کمتر**
  - اگر مشکل سرعت است و ظرفیت دارید: parallelism را کمی بالا ببرید
- **نکتهٔ یوتیوب / googlevideo**:
  - برای URLهای `*.googlevideo.com` که `clen=` دارند، موتور به‌صورت خودکار چانک را بزرگ‌تر و همزمانی را محدودتر می‌کند تا تعداد فراخوانی‌های Apps Script کمتر شود.

### `relay_request_timeout_secs`

- **چی هست؟** timeout هر تماس Apps Script.
- **اثر روی UX**:
  - خیلی کم: timeout کاذب زیاد → retry زیاد → «خراب به نظر می‌رسد»
  - خیلی زیاد: گیرها طولانی‌تر حس می‌شوند → UI انگار stuck است
- **کی تغییر بدهم؟**
  - اگر رله کار می‌کند ولی کند است، کمی بالا ببرید.
  - پایین آوردن فقط وقتی منطقی است که چند `script_id` دارید و می‌خواهید failover سریع‌تر شود.

## شبکه / دسترسی

### `sni_hosts`

- **چی هست؟** لیست چرخش SNI برای اتصال TLS به `google_ip`.
- **اثر روی UX**:
  - pool خوب → عبورپذیری بیشتر در برابر فیلتر
  - یک SNI محلی-بلاک‌شده اگر داخل pool باشد → timeout‌های تصادفی
- **کی تغییر بدهم؟**
  - اگر تستر SNI نشان می‌دهد بعضی نام‌ها همیشه fail هستند، لیست دستی بسازید و آن‌ها را حذف کنید.

### `fetch_ips_from_api` + `max_ips_to_scan` + `scan_batch_size` + `google_ip_validation`

- **چی هستند؟** کنترل نحوهٔ پیدا کردن/اسکن `google_ip` (منبع IPها، اندازهٔ اسکن، و چک‌های ایمنی).
- **اثر روی UX**:
  - اسکن بزرگ‌تر → شانس یافتن IP سریع‌تر، ولی زمان/مصرف شبکه بیشتر
  - validation کمک می‌کند `google_ip` اشتباه (غیر گوگل) باعث fail نشود

## مسیریابی (وقتی یک سایت می‌شکند)

### `passthrough_hosts`

- **چی هست؟** برای این هاست‌ها Apps Script را دور می‌زند (TCP مستقیم، و اگر `upstream_socks5` دارید از آن).
- **اثر روی UX**:
  - quota کمتر مصرف می‌شود و MITM برای آن سایت حذف می‌شود
  - ولی اگر آن سایت روی شبکهٔ شما مستقیم بلاک باشد، دیگر bypass ندارد

### `domain_overrides`

- **چی هست؟** قوانین per-host برای مجبور کردن مسیر (`direct` / `sni_rewrite` / `relay` / `full_tunnel`) و گزینهٔ `never_chunk`.
- **اثر روی UX**:
  - بهترین راه برای «فقط یک سایت را درست کن» بدون بهم‌ریختن کل سیستم
  - `never_chunk=true` برای سایت‌های حساس/ضدبات که با chunking مشکل دارند مفید است

### `hosts`

- **چی هست؟** نگاشت میزبان‌ها برای تصمیم مسیر (گوگل-اِج مستقیم vs Apps Script).
- **اثر روی UX**:
  - اگر یک دامنه را روی مسیر گوگل-اِج بگذارید ممکن است سریع‌تر شود و مشکل User-Agent ثابت Apps Script را نداشته باشد
  - ولی فقط برای میزبان‌هایی معنی دارد که از مسیر SNI-rewrite قابل دسترسی‌اند

### `normalize_x_graphql`

- **چی هست؟** کوتاه‌کردن queryهای نویزی GraphQL در X برای افزایش cache hit.
- **اثر روی UX**:
  - معمولاً X بسیار روان‌تر می‌شود (کَش بیشتر، تماس رله کمتر)
  - در موارد کمی ممکن است endpointها به پارامترهای حذف‌شده حساس باشند → اگر چیزی شکست خاموشش کنید

### `youtube_via_relay`

- **چی هست؟** یوتیوب را به جای مسیر مستقیم SNI-rewrite از Apps Script عبور می‌دهد.
- **اثر روی UX**:
  - می‌تواند Restricted Mode/SafeSearch-on-SNI را دور بزند
  - quota مصرف می‌کند و برای یوتیوب هم User-Agent ثابت Apps Script را اعمال می‌کند

## چند-اکانت و مقاومت در برابر quota

### `account_groups[]`

- **چی هست؟** تعریف چند «pool سهمیه» مستقل (معمولاً هر گروه یک اکانت گوگل + چند deployment).
- **اثر روی UX**:
  - گروه‌های بیشتر + `script_id`های بیشتر → ظرفیت بیشتر، fail کمتر، 504 کمتر
  - `weight` کمک می‌کند یکی را ترجیح دهید و بقیه بکاپ باشند

---

## ایمنی (گوش دادن روی LAN)

### `listen_host` + `lan_token` + `lan_allowlist`

- **چی هست؟** تعیین می‌کند پروکسی فقط روی localhost باشد یا روی LAN هم گوش بدهد و چه گاردریل‌هایی فعال شوند.
- **اثر روی UX**:
  - LAN راحت است برای استفاده از دستگاه‌های دیگر، ولی ریسک را بالا می‌برد
  - `lan_token` یک توکن ساده برای HTTP/CONNECT می‌گذارد
  - `lan_allowlist` مهم‌تر برای SOCKS5 است (چون هدر توکن ندارد)

---

## خودترمیم / پایداری

### `outage_reset_*`

- **چی هست؟** اگر در یک پنجرهٔ کوتاه چند خطای «معتبر» رخ دهد، pool اتصال keep-alive ریست می‌شود تا از مسیر poisoned خارج شود، سپس cooldown می‌گذارد.
- **اثر روی UX**:
  - می‌تواند «تا ریستارت نشه درست نمی‌شه» را به ریکاوری خودکار تبدیل کند
  - اگر خیلی تهاجمی تنظیم شود، churn و reconnect اضافی ایجاد می‌کند
